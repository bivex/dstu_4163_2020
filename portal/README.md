# Портал підписання та редагування документів

Бекенд-скелет порталу поверх доменного ядра **dilovod4** (ДСТУ 4163:2020 + 7 НПА).
Багатопідписання, генерація/валідація документів, черга підписів, аудит зберігання.

## Стек

- **Бекенд:** FastAPI + SQLAlchemy 2.0 поверх `dilovod4` (Python-домен — генерація
  PDF/DOCX, валідація за ДСТУ 4163 та правила ст.7/21 НПА).
- **БД:** SQLite на томі (за замовчуванням) або PostgreSQL (продакшн — див.
  `docker-compose.yml`).
- **Підпис КЕП / печатка юрособи:** на КЛІЄНТІ (EUSign, `external/EUSignES6`) —
  приватний ключ не покидає браузер (Закон 2155-VIII). Сервер приймає готовий
  p7s/asice і сам визначає тип сертифіката (eSign — КЕП особи, чи eSeal —
  електронна печатка юрособи/ФОП) за QC statements (ETSI EN 319 412-5).
  - **Файловий ключ** (.dat/.pfx/.jks) — працює з WASM-збіркою EUSign у репо.
  - **Апаратний токен** (е.Ключ, Алмаз, ID-картка) — потребує локально
    встановленого «ІІТ Користувач ЦСК» (native host euscpnmh). Фронт робить
    feature-detection: якщо токен-здатний модуль активний — показує перелік
    носіїв і підписує токеном; інакше — чітке повідомлення + файловий режим.
  - **Електронна печатка юрособи (eSeal)** — той самий DSTU 4145 / CAdES-X-Long,
    але сертифікат печатки (QC type eSeal, subject = назва юрособи,
    organizationIdentifier = NTRUA-ЄДРПОУ). Підписант зі `signer_type="seal"`
    у черзі; авторизація — за збігом CN сертифіката печатки, прив'язаного в
    кабінеті користувача (`organization_cert_cn`).
- **Фронт:** Nuxt 4 + Nuxt UI 4 (Vue 3, підмодуль `external/dms-dir`); у dev ходить
  на `api:8000`. Маршрут дашборду підпису — `/dashboard`, EUSign WASM динамічно
  імпортується з порталу через проксі `/api/eusign/**`.
- **Серверний підпис** токеном/печаткою юрособи — `dilovod4.token_sign`/`asic`
  (КНЕДП ДПС), підключається окремим маршрутом за потреби.

### Тестові сертифікати

```bash
# згенерувати CA + КЕП особи (eSign) + печатку юрособи (eSeal)
python scripts/gen_test_certs.py --out samples/test_certs/
# пароль PKCS#12: testpassword
```
Сертифікати — криптографічно валідні ECDSA P-256 з українськими QC-розширеннями
(сумісні з EUSign/openssl/UAPKI). Генератор: `dilovod4.infrastructure.test_cert_factory`.

### Серверний підпис печаткою юрособи

`POST /documents/{doc_id}/server-seal` — сервер САМ генерує CAdES-підпис печаткою
(ключ у PKCS#12 на сервері), без участі браузера. Зручно для масового підписання
(пакет наказів, розрахункових документів). Авторизація: юзер з прив'язаною печаткою
(`organization_cert_cn` == CN підписанта-печатки), як і клієнтський підпис печаткою.

Налаштування через env:

| Змінна | Призначення |
|--------|-------------|
| `PORTAL_SEAL_P12` | шлях до PKCS#12 печатки юрособи (обовʼязкове; без нього ендпоїнт → 503) |
| `PORTAL_SEAL_PASSWORD` | пароль контейнера (з env, не в коді) |
| `PORTAL_SEAL_CERT_CACHE` | кеш сертифікатів (типово `../.euscp_store`) |
| `PORTAL_SEAL_CRL_CACHE` | кеш CRL (типово `/tmp/crls`) |
| `PORTAL_SEAL_WITH_TIMESTAMP=1` | CAdES-T з квал. позначкою часу (Art.26.4) |
| `PORTAL_SEAL_TSP_URL` / `PORTAL_SEAL_CMP_URL` | TSP/CMP-адреси КНЕДП (для CAdES-T) |

Генерація тестового PKCS#12 печатки (для локальної розробки):

```bash
python scripts/gen_seal_p12.py --org "ТОВ Моя Організація" --edrpou 12345678 \
    --out /etc/dilovod/seal.p12 --password 's3cret'
# потім:
export PORTAL_SEAL_P12=/etc/dilovod/seal.p12
export PORTAL_SEAL_PASSWORD='s3cret'
```

⚠ **Безпека**: ключ печатки у файлі — для захищеного сервера (0600, окремий
користувач-сервіс). Пароль лише з env, ніколи не логується. Для продакшну з
високими вимогами — HSM/PKCS#11 (окрема інтеграція). Ключ печатки юрособи —
LEGAL для серверного зберігання (на відміну від КЕП особи, що вимагає
«ключ не покидає браузер» за ЗУ 2155-VIII): печатка належить юрособі, тож сервер
як її представник може підписувати нею. Адаптер: `dilovod4.infrastructure.server_seal`.

## Запуск

```bash
cd portal
docker compose up -d
curl http://localhost:8000/health
```

Документація API: http://localhost:8000/docs (Swagger).

## Тести

```bash
PYTHONPATH=src python -m pytest portal/tests/ -q
```

20 тестів: повний цикл багатопідписання, черга за порядком, валідація
ДСТУ/НПА (15 правил), генерація/завантаження PDF, аудит, edge-cases
(підпис поза чергою → 409, редагування після submit → 409 тощо).
Кожен тест отримує ізольований SQLite через фікстуру.

## Життєвий цикл документа (перевірено e2e)

```
create (DRAFT, черга підписантів)
  → generate  (PDF/DOCX + валідація ДСТУ/НПА, 15 правил)
  → submit    (PENDING_SIGNATURES, перший підписант → INVITED)
  → sign #0   (директор; КЕП з клієнта; наступний → INVITED)
  → sign #1   (головбух; усі підписали → SIGNED)
  → publish   (PUBLISHED, ст.14 996-XIV / ст.15 2939-VI)
```

Черга суворо за порядком: підпис поза чергою → HTTP 409.

## Ендпоінти

| Метод | Шлях | Призначення |
|-------|------|-------------|
| GET   | `/health` | стан сервісу |
| POST  | `/documents` | створити чернетку + чергу підписантів |
| GET   | `/documents` | перелік документів |
| GET   | `/documents/{id}` | документ + підписанти + аудит + звіт |
| PUT   | `/documents/{id}` | редагувати чернетку (лише DRAFT) |
| POST  | `/documents/{id}/generate` | згенерувати PDF/DOCX + валідація |
| GET   | `/documents/{id}/download` | завантажити згенерований файл |
| POST  | `/documents/{id}/validate` | перевірка за ДСТУ/НПА (без генерації) |
| POST  | `/documents/{id}/submit` | у чергу підписання |
| POST  | `/documents/{id}/sign` | прийняти КЕП-підпис від клієнта |
| POST  | `/documents/{id}/reject` | відмова підписанта |
| POST  | `/documents/{id}/publish` | оприлюднити підписаний документ |

## Відповідність НПА

- **851-IV** — підпис КЕП завершує створення (ст.6); статус оригіналу (ст.7);
  зберігання `retention_until` ≥ паперового (ст.13).
- **80/94-ВР** — self-hosted обробка в авторизованій системі (ст.8).
- **2657-XII / 2939-VI** — режим доступу, оприлюднення.
- **996-XIV** — реквізити первинного документа, строк оприлюднення.
