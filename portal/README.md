# Портал підписання та редагування документів

Бекенд-скелет порталу поверх доменного ядра **dilovod4** (ДСТУ 4163:2020 + 7 НПА).
Багатопідписання, генерація/валідація документів, черга підписів, аудит зберігання.

## Стек

- **Бекенд:** FastAPI + SQLAlchemy 2.0 поверх `dilovod4` (Python-домен — генерація
  PDF/DOCX, валідація за ДСТУ 4163 та правила ст.7/21 НПА).
- **БД:** SQLite на томі (за замовчуванням) або PostgreSQL (продакшн — див.
  `docker-compose.yml`).
- **Підпис КЕП:** на КЛІЄНТІ (EUSign, `external/EUSignES6`) — приватний ключ не
  покидає браузер (Закон 2155-VIII). Сервер приймає готовий p7s/asice.
- **Фронт:** Next.js 15 + React 19 (база — `external/SA-SignInfo`), підключається
  окремо; у dev ходить на `api:8000`.
- **Серверний підпис** токеном/печаткою юрособи — `dilovod4.token_sign`/`asic`
  (КНЕДП ДПС), підключається окремим маршрутом за потреби.

## Запуск

```bash
cd portal
docker compose up -d
curl http://localhost:8000/health
```

Документація API: http://localhost:8000/docs (Swagger).

## Життєвий цикл документа (перевірено e2e)

```
create (DRAFT, черга підписантів)
  → generate  (PDF/DOCX + валідація ДСТУ/НПА, 15 правил)
  → submit    (PENDING_SIGNATURES, перший підписант → INVITED)
  → sign #0   (директор; КЕП з клієнта; наступний → INVITED)
  → sign #1   (головбух; усі підписали → SIGNED)
  → publish   (PUBLISHED, ст.14 996-XIV / ст.15 2939-VI)
```

Черга суворо за порядком: підпис поза чергою → HTTP 409.

## Ендпоінти

| Метод | Шлях | Призначення |
|-------|------|-------------|
| GET   | `/health` | стан сервісу |
| POST  | `/documents` | створити чернетку + чергу підписантів |
| GET   | `/documents` | перелік документів |
| GET   | `/documents/{id}` | документ + підписанти + аудит + звіт |
| PUT   | `/documents/{id}` | редагувати чернетку (лише DRAFT) |
| POST  | `/documents/{id}/generate` | згенерувати PDF/DOCX + валідація |
| GET   | `/documents/{id}/download` | завантажити згенерований файл |
| POST  | `/documents/{id}/validate` | перевірка за ДСТУ/НПА (без генерації) |
| POST  | `/documents/{id}/submit` | у чергу підписання |
| POST  | `/documents/{id}/sign` | прийняти КЕП-підпис від клієнта |
| POST  | `/documents/{id}/reject` | відмова підписанта |
| POST  | `/documents/{id}/publish` | оприлюднити підписаний документ |

## Відповідність НПА

- **851-IV** — підпис КЕП завершує створення (ст.6); статус оригіналу (ст.7);
  зберігання `retention_until` ≥ паперового (ст.13).
- **80/94-ВР** — self-hosted обробка в авторизованій системі (ст.8).
- **2657-XII / 2939-VI** — режим доступу, оприлюднення.
- **996-XIV** — реквізити первинного документа, строк оприлюднення.
