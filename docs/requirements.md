# Dilovod4 — вимоги та карта відповідності ДСТУ 4163:2020

## 1. Бізнес-мета

Дати юридичним особам інструмент, що **детермін101овано перевіряє** оформлення
організаційно-розпорядчого документа на відповідність ДСТУ 4163:2020 і повертає
структурований звіт із конкретними порушеннями (а не «так/ні»).

Двигун призначений для повторного використання: CLI сьогодні, веб-API / інтеграція
з СЕД завтра — без зміни доменної логіки.

## 2. Функціональні вимоги

- FR-1. Прийняти опис документа (реквізити, геометрія, типографіка, метадані).
- FR-2. Перевірити структурну валідність (інваріанти) до будь-яких правил.
- FR-3. Застосувати набір правил відповідності ДСТУ і зібрати звіт.
- FR-4. Кожне порушення має: код правила, посилання на параграф, повідомлення.
- FR-5. Звіт має агрегований підсумок `conforms` + перелік знахідок.
- FR-6. Можливість вмикати/вимикати окремі правила (профіль перевірки).
- FR-7. CLI приймає JSON-опис документа і друкує звіт (текст або JSON).

## 3. Нефункціональні вимоги

- NFR-1. Домен не залежить від БД, мережі, UI, фреймворків (чистий Python).
- NFR-2. Правила детерміновані та чисті (без IO, без глобального стану).
- NFR-3. Контракти портів стабільні; нові правила додаються без зміни ядра (OCP).
- NFR-4. Конфігурація (профіль, формат виводу) — через env / аргументи, не в коді.
- NFR-5. Помилки домену відрізняються від технічних збоїв на рівні типів.
- NFR-6. Домен покрито unit-тестами; межі (адаптери) — інтеграційними.
- NFR-7. Структурне логування ключових подій на межах застосунку.

## 4. Карта відповідності: параграф ДСТУ → Catala scope → доменне правило

| § | Catala scope | Доменне правило (`rule_id`) | Що перевіряє |
|---|---|---|---|
| 4.4 | `MandatoryRequisites` | `MANDATORY_REQUISITES` | Наявність обовʼязкових реквізитів; підпис (паперовий/електронний). |
| 6.1/6.2/6.5 | `DocumentGeometry` | `DOCUMENT_GEOMETRY` | Формат, поля, допуск розташування ±2 мм. |
| 6.7 | `RequisiteFormatting` | `REQUISITE_FORMATTING` | Вирівнювання; заборона підкреслення/рисок. |
| 7.2/7.6 | `Typography` | `TYPOGRAPHY` | Гарнітура, кеглі, довжина рядка ≤28 знаків. |
| 7.3 | `LineSpacing` | `LINE_SPACING` | Міжрядковий інтервал за форматом. |
| 7.7 | `LeftIndentation` | `LEFT_INDENTATION` | Відступи 10/90/100/100/125 мм. |
| 7.10 | `PageNumbering` | `PAGE_NUMBERING` | Нумерація 2+ сторінок, без слова «сторінка». |
| 7.11 | `PrintSideByRetention` | `PRINT_SIDE` | Бік друку за строком зберігання. |
| 5.15 | `AddresseeRule` | `ADDRESSEE` | ≤4 адресати, інакше список розсилання. |
| 5.21 | `AppendixRule` | `APPENDIX` | >10 додатків → опис. |
| 6.8/6.9 | `BlankRequisites` | `BLANK_REQUISITES` | Дозволені коди реквізитів; поріг 2000/рік. |
| 5.10 | `DateRule` | `DATE` | Спосіб дати; спільний документ — одна дата. |
| 5.1/5.10/5.31 | `SymbolDimensions` | `SYMBOL_DIMENSIONS` | Герб 17×12, емблема ≤17, QR 21×21, зона 60×100. |

## 5. Поза межами (як і в Catala)

Координатні схеми (Додаток А), зразки бланків (Додаток Б), приклади документів,
бібліографія, повна граматика реквізитів — не кодуються.
